---
description: Cut a release from main. Aggregates pending changesets into CHANGELOG.md, bumps the version, tags it, and optionally creates a GitHub Release.
argument-hint: [optional pre-release tag, e.g. "rc.1" or "beta.0"]
---

Cut a new release directly on `main`. Aggregates all `.changeset/*.md` files accumulated since the last release into a versioned `CHANGELOG.md` entry, bumps the package version (and the plugin manifest), commits, tags `vX.Y.Z`, pushes, and optionally publishes a GitHub Release.

This command assumes [GitHub Flow](https://docs.github.com/en/get-started/using-github/github-flow) (`feature/* → main`, releases cut from `main` via tags) and [`@changesets/cli`](https://github.com/changesets/changesets). There is no `develop` branch, no `release/*` branch, and no back-merge: `main` is the single trunk, and a release is just a tagged commit on it.

The release commit is the **one sanctioned exception** to the "everything via PR" rule — it's mechanical, generated by `changeset version`, and reviewed in-terminal at step 6 before it's pushed.

## Procedure

1. **Gather context.** Run in parallel:
   - `git branch --show-current` — must be `main`. Refuse otherwise.
   - `git status` (no `-uall`) — must be clean.
   - `git fetch origin main --quiet` — refresh refs.
   - `git rev-list --left-right --count origin/main...HEAD` — local must be in sync with `origin/main` (no ahead, no behind).
   - `ls .changeset/*.md 2>/dev/null | grep -v README` — must list at least one changeset.
   - `cat package.json | grep '"version"'` — current version, the floor for the bump.
   - `gh auth status` — `gh` must be authenticated (needed only if you create a GitHub Release).

2. **Refuse to proceed if preconditions fail.** Stop and tell the user:
   - Not on `main` → `git checkout main && git pull`.
   - Working tree dirty → commit or stash first.
   - Out of sync with `origin/main` → `git pull --ff-only` (or rebase if diverged). If local is *ahead*, those are unreleased commits that should have gone through a PR — confirm with the user before releasing them.
   - No changesets → there is nothing to release; tell the user and stop.

3. **Show the release plan and get explicit confirmation.** Print:
   - Current version (from `package.json`).
   - List of pending changesets — for each, the bump type and the one-line summary.
   - Computed next version (use `npx @changesets/cli status` to compute; or describe the rule: `major` wins over `minor` wins over `patch`).
   - The pre-release tag from `$ARGUMENTS`, if provided (e.g., `1.4.0-rc.1`).

   Ask: *"Proceed with release vX.Y.Z?"* — wait for `yes`. Do not proceed silently.

4. **Apply the version bump and aggregate the changelog.** Run on `main` directly — no release branch.

   - **Stable release:**
     ```bash
     npx @changesets/cli version
     ```
   - **Pre-release** (if `$ARGUMENTS` was provided):
     ```bash
     npx @changesets/cli pre enter <tag>
     npx @changesets/cli version
     # `pre exit` happens at the next stable release, not now
     ```

   This will:
   - Update `"version"` in `package.json`.
   - Move all `.changeset/*.md` entries into a new section at the top of `CHANGELOG.md`, grouped by `Major Changes` / `Minor Changes` / `Patch Changes` (Changesets' default — see Notes below).
   - Delete the consumed `.changeset/*.md` files.

5. **Sync the plugin manifest version.** `package.json` is the source of truth; the Claude Code plugin manifest carries its own `version` the marketplace UI reads, so keep them in lockstep:

   ```bash
   ./scripts/sync-plugin-version.sh
   ```

   The script copies `package.json`'s version into `plugins/gcenv/.claude-plugin/plugin.json` (idempotent — a no-op if already equal). If the project grows other parallel version fields (a `VERSION` file, a Cargo/PyPI manifest), update them here too.

6. **Show the diff to the developer for review.** Print:
   - `git diff --stat`
   - The full new section added to `CHANGELOG.md`
   - The version delta (`X.Y.Z` → `X'.Y'.Z'`)

   Ask: *"CHANGELOG looks right? Proceed to commit, tag, and push?"* — wait for confirmation. This in-terminal review **replaces** the old release-PR eyeball step; the developer may want to hand-edit the changelog prose before it ships (this is the moment).

7. **Commit, tag, and push.** Conventional commit subject so downstream tooling can recognize it. Use an **annotated** tag (carries metadata; required for `git describe`):

   ```bash
   git add CHANGELOG.md package.json plugins/gcenv/.claude-plugin/plugin.json .changeset/
   git commit -m "$(cat <<'EOF'
   chore(release): v<X.Y.Z>

   Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>
   EOF
   )"
   git tag -a v<X.Y.Z> -m "Release v<X.Y.Z>"
   git push origin main
   git push origin v<X.Y.Z>
   ```

   Never `--no-verify`. Never `--force`. If `git push origin main` is rejected by branch protection (a rule that forbids direct pushes to `main`), stop and tell the user — they'll need to land the release commit via a short-lived PR instead. Don't try to force it.

8. **Optionally create a GitHub Release.** A git tag already marks the version; a GitHub Release adds a public, notified notes page (and a home for artifacts, of which this repo has none). Ask the user: *"Create a GitHub Release for v<X.Y.Z>?"* If yes:

   ```bash
   gh release create v<X.Y.Z> --title "v<X.Y.Z>" --notes "$(cat <<'EOF'
   <paste the new CHANGELOG.md section verbatim, minus the version heading>
   EOF
   )"
   ```

   For a pre-release, add `--prerelease`. If the user declines, the tag still stands on its own — skip this step.

9. **Report.** Print the new version, the tag, the count of changesets consumed, and the GitHub Release URL if one was created.

## Anti-patterns — do not do

- **Do not run `/create-release` from any branch other than `main`.**
- **Do not edit `CHANGELOG.md` by hand before running `changeset version`.** The tool rewrites the file; manual edits made before will be clobbered. Hand-edit *after* step 4, before step 7.
- **Do not delete `.changeset/README.md`** if Changesets created it — it documents the format for new contributors. `changeset version` only consumes `*.md` files that have frontmatter.
- **Do not tag before the release commit exists.** The tag must point at the `chore(release)` commit, so tag *after* committing (step 7), not before.
- **Do not skip step 6 (the human review step).** With no release PR, the in-terminal review is the only point a human eyeballs the changelog before it ships.
- **Do not auto-publish to a registry from this command.** This repo ships nothing to a registry (it's installed from source via the marketplace). If that ever changes, publishing belongs in CI on tag push, not in an interactive command on a laptop.

## Notes on the heading style

Changesets' default `CHANGELOG.md` uses `### Major Changes` / `### Minor Changes` / `### Patch Changes`. [Keep a Changelog 1.1.0](https://keepachangelog.com/en/1.1.0/) prefers `### Added` / `### Changed` / `### Deprecated` / `### Removed` / `### Fixed` / `### Security`. They are not directly compatible. Pick one and stick to it. To use the Keep-a-Changelog headings, configure `"changelog"` in `.changeset/config.json` to a custom formatter. The default is fine for most projects and is what this command assumes.
